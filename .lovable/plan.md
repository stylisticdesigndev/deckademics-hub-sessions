## Goal

Stop the Admin → Messages page from showing automated student-to-instructor notifications ("Marked Absent", "Absence Cancelled", "Running Late") that are intended for instructors only.

## Root cause

The admin account also holds the `instructor` role, so `session.user.id` matches the `receiver_id` of automated messages students send to their instructor. `AdminMessages.tsx` fetches every message where the user is sender or receiver, so instructor-side automation leaks into the admin view.

## Approach (Option A — frontend filter)

Filter out automated/system messages from the admin Messages view by their well-known `subject` values. These three subjects are the only ones generated by the automation flows:

- `Running Late` (src/components/student/RunningLateButton.tsx:43)
- `Marked Absent` (src/hooks/student/useStudentClassAttendance.ts:274)
- `Absence Cancelled` (src/hooks/student/useStudentClassAttendance.ts:364)

## Changes

**`src/pages/admin/AdminMessages.tsx`**

1. After fetching `messages`, filter out any message whose `subject` is one of the three automated subjects above before storing into `allMessages`. Use a constant set:
   ```ts
   const AUTOMATED_SUBJECTS = new Set(['Running Late', 'Marked Absent', 'Absence Cancelled']);
   ```
2. Apply the same filter when grouping conversations so that a conversation built entirely from automated messages disappears from the admin Conversations list.
3. Manual messages composed by/to the admin remain unaffected.

No DB schema changes, no migrations, no edge-function changes. Instructor and student views are untouched and continue to receive these notifications normally.

## Acceptance criteria

- Admin Messages page no longer lists Dennis Coffey or Son Gohan threads composed only of absence/late notifications.
- Admin can still see and reply to manual conversations they originated or were directly addressed in.
- Instructor Messages view still shows the late/absence notifications as before.
